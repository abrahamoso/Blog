My First API Project: How I Got Started in Software Development

December 13, 2024

Starting something new is always intimidating, and my first API project was no exception. I dove headfirst into a project that combined healthcare and technology: building an FDA Drug Information API. Like most beginners, I started with a blank screen and a million questions. The enormity of the project quickly became clear: I couldn’t do it alone. What followed was a journey of mentorship, trial and error, and some invaluable lessons about growth and resilience. Here’s what I learned—and what you can take away if you’re just getting started, too.

Finding A Mentor

If you’re new to coding, one of the best things you can do is find a mentor. That’s what I did through "Mentorcruise," a platform that connects aspiring developers with experienced engineers. At first, I hesitated—imposter syndrome made me question whether I was even ready to ask for help. Scrolling through profiles of engineers from top tech companies, I thought, "No way I should even try this, right?" But after weeks of self-doubt, I finally took the plunge and connected with a mentor. It was the best decision I could’ve made. Through mentoring sessions, I learned about essential tools like Git, how to structure a project, and how to handle feedback without feeling discouraged. If you’re considering mentorship, don’t wait—it’s an investment in yourself that pays off in ways you can’t imagine.

Overcoming Technical Challenges

One of the biggest challenges of starting this project was feeling completely out of my depth. I vividly remember one of my first mentoring sessions when my mentor asked if I knew how to use Git. When I admitted I didn’t, he patiently walked me through the basics of version control. At the time, I interpreted his directness as frustration with my lack of knowledge. But looking back, his straightforward approach was exactly what I needed. It taught me that the key to growth is embracing the discomfort of learning and being open to feedback—even when it stings. This project forced me to confront a lot of my fears about starting something new, especially the fear of looking foolish. I learned that it’s okay not to know everything right away. What matters most is showing up, asking questions, and taking that first step.

For example:

  • Learning to work better with multiple APIs and handle JSON responses
  • Implementing proper error handling for user inputs
  • Creating a user-friendly interface for displaying complex medical information

Key Lessons for Beginners

One of the most valuable lessons I learned from this project was how to work with APIs.

What is an API?

For those unfamiliar, an API (Application Programming Interface) is like a bridge that allows two software systems to communicate. Imagine you’re at a restaurant: you (the user) place an order with the waiter (the API), who communicates with the kitchen (the server). The kitchen prepares the food (the data) and the waiter delivers it back to you. APIs are everywhere in modern apps, from weather forecasts to social media. For this project, I worked with the FDA’s API to retrieve drug information.

Learning Postman Basics

To test API requests, I used a tool called Postman. It became an essential part of my workflow, acting as a sandbox where I could experiment with API calls and responses. Here’s what I learned to do:

  • Send GET requests to retrieve data.
  • Use POST requests to send new to a server.
  • Experiment with DELETE requests to remove data.

While I’m not perfect at using Postman yet, I now understand the basics. This tool helped me bridge the gap between theory and practical application, making APIs feel far less intimidating.

Future Improvements and Next Steps

Though I’ve moved on from this project, reflecting on it has helped me shape my future goals. Here’s what I’ll take into my next projects:

  • Prioritizing Code Organization: Starting with a plan for clean, efficient code saves time in the long run.
  • Focusing on User Experience: I’ll prioritize creating interfaces that make complex data easy to understand.
  • Mastering API Workflows: I want to explore advanced API integration techniques, like authentication and handling complex responses.
